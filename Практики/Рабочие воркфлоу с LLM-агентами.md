# Рабочие воркфлоу с LLM-агентами

> Практические workflow для разработки, дебага, code review и проектной работы. Anti-patterns.

Назад: [[LLM Wiki]] | Папка: `Практики/` | Связанные: [[Prompt Engineering для разработки]], [[Claude Code]], [[GSD (Get Shit Done)]]

---

## 1. Три режима работы -- выбирай правильный ([best practices](https://code.claude.com/docs/en/best-practices))

Прежде чем начинать -- определи тип задачи. От этого зависит весь workflow.

### Routine tasks (Agent-First)
**Когда**: 95% работы -- знакомые паттерны, CRUD, типовые фичи.
- Используй Plan Mode -- опиши цель, дай агенту построить план
- Итерируй 1-3 раза ("Keep planning", "А что насчёт edge cases?")
- Только потом переключай в execution

### Specialized tasks (Human-First)
**Когда**: нетривиальная логика, интеграция, миграция данных.
- Разбивай на маленькие шаги с human review между ними
- **Коммить после каждого успешного шага** -- это критично
- Агенты часто "хорошо начинают, плохо заканчивают", и без коммитов получишь кашу
- Пропусти Plan Mode; используй `git diff` для review

### Novel systems (Manual + AI assistance)
**Когда**: новая архитектура, критические подсистемы, production groundwork.
- Пиши сам, используй LLM только для вопросов и сниппетов
- Ты должен полностью понимать каждую строку
- Агент тут помощник, не исполнитель

## 2. Workflow: новая фича от начала до конца

### Шаг 1: Спецификация (Plan Mode)

```
Нужно реализовать систему нотификаций для приложения.

Требования:
- Email и in-app уведомления
- Настраиваемые preferences для каждого пользователя
- Поддержка templates
- Очередь через Redis/BullMQ

Прочитай текущую архитектуру (agent_docs/architecture.md)
и предложи план реализации. Не пиши код пока.
```

Итерируй: "А как с retry logic?", "Покажи схему БД", "А что с rate limiting?"

### Шаг 2: Декомпозиция на подзадачи

После согласования плана -- разбей на атомарные шаги. Используй `/todos` (Ctrl+T) чтобы не терять фокус:

```
Разбей план на подзадачи для последовательной реализации.
Каждая подзадача должна быть самодостаточной и коммитабельной.
Начни с data layer, потом service layer, потом API, потом UI.
```

### Шаг 3: Реализация по шагам

Для каждой подзадачи:

```
Реализуй [подзадача N]: создание Prisma-схемы для notifications.
Следуй плану из предыдущего шага.
После реализации: запусти typecheck и тесты.
```

**После каждого шага:**
1. Review через `git diff`
2. Если ОК --> `git commit`
3. `/clear`
4. Следующая подзадача

### Шаг 4: Интеграционное тестирование

```
Прочитай все изменения в ветке (git diff main...HEAD).
Напиши интеграционные тесты для системы нотификаций.
Покрой основные сценарии:
- Создание и отправка email-нотификации
- In-app нотификация с preferences check
- Retry при ошибке отправки
- Rate limiting
```

### Шаг 5: Review и доводка

```
Проведи финальный code review всех изменений в ветке.
Фокус: edge cases, error handling, production readiness.
```

## 3. Workflow: Debug

### Быстрый debug (1 промпт)

```
Ошибка: POST /api/orders возвращает 500.

Error log:
[paste error trace]

Последние изменения: git log --oneline -5
Воспроизведение: curl -X POST localhost:3000/api/orders -d '{"userId": 1}'

Найди причину и предложи фикс.
```

### Глубокий debug (multi-step)

**Шаг 1: Сбор информации**
```
Пользователи жалуются на медленную загрузку страницы /dashboard.
Среднее время: 8-12 секунд (должно быть <2 сек).

Исследуй:
1. Какие API вызовы делает dashboard (прочитай компонент)
2. Есть ли N+1 queries в API handlers
3. Проверь наличие индексов для используемых queries
Пока не фикси -- только собери данные.
```

**Шаг 2: Анализ и план**
```
На основе собранных данных -- какая root cause?
Предложи план исправления, ранжируй по impact.
```

**Шаг 3: Исправление**
```
Реализуй фикс N1 (наибольший impact).
Напиши тест, подтверждающий улучшение.
```

### Использование sub-agent для тестов

Вместо заливки всего test output в основной контекст:

```yaml
---
name: test-runner
description: 'Run tests and return only failures with root cause analysis'
model: sonnet
---
Запусти тесты командой: pnpm test
Верни ТОЛЬКО:
- Список упавших тестов
- Error message для каждого
- Предполагаемая причина
Если все тесты прошли -- скажи "All N tests passed".
```

## 4. Workflow: Code Review

### Self-review перед PR

```
Проведи code review моих изменений (git diff main...HEAD).

Checklist:
- [ ] Нет hardcoded значений
- [ ] Error handling для всех внешних вызовов
- [ ] Нет секретов в коде
- [ ] Тесты покрывают happy path + основные edge cases
- [ ] Нет breaking changes в API
- [ ] Типы корректны (нет any, as unknown)

Формат: CRITICAL / WARNING / SUGGESTION / OK для каждого пункта.
```

### Review чужого PR

```
Прочитай PR #123 (gh pr view 123 --json body,diff).

Контекст проекта: [кратко или сошлись на CLAUDE.md]

Оцени:
1. Решает ли PR заявленную проблему?
2. Есть ли баги или edge cases?
3. Соответствует ли архитектурным решениям проекта?
4. Достаточно ли тестов?

Напиши review comments в формате, который можно вставить в GitHub.
```

## 5. Workflow: Проектная работа

### Планирование архитектуры

```
Мы проектируем систему real-time аналитики.

Требования:
- 10k events/sec ingestion
- Агрегации за 1 мин / 1 час / 1 день
- Dashboard с обновлением раз в 5 сек
- Retention: raw 7 дней, агрегированные 1 год

Текущий стек: Node.js, PostgreSQL, Redis, Kubernetes.

Предложи 2-3 варианта архитектуры. Для каждого:
- Компоненты и их взаимодействие
- Плюсы и минусы
- Оценка сложности реализации
- Какие новые технологии потребуются
```

### Создание технической документации

```
Создай ADR (Architecture Decision Record) для решения
использовать ClickHouse для аналитики.

Структура:
- Контекст: какую проблему решаем
- Рассмотренные варианты (PostgreSQL, TimescaleDB, ClickHouse)
- Решение и обоснование
- Последствия: что это меняет в архитектуре
- Статус: Proposed

Основывайся на наших обсуждениях и коде в src/analytics/.
```

### Онбординг документация

```
Изучи структуру проекта и напиши developer onboarding guide.

Покрой:
1. Как поднять dev environment
2. Структура проекта -- что где лежит
3. Основные workflows: как добавить API endpoint, как написать тест
4. Ключевые архитектурные решения (из agent_docs/architecture.md)
5. FAQ: частые проблемы при разработке
```

## 6. Комбинирование инструментов ([Habr: субагенты](https://habr.com/ru/articles/946748/) | [Habr: контекст](https://habr.com/ru/articles/971772/))

### Claude Code + MCP servers

**Типичный набор MCP:**

| MCP Server | Назначение | Когда включать |
|---|---|---|
| context7 | Актуальная документация библиотек | Всегда для проекта |
| claude-mem | Persistent memory между сессиями | Всегда |
| GitHub MCP | Работа с issues, PRs | При работе с GitHub |
| Database MCP | Прямые запросы к БД | При дебаге данных |
| Browser MCP | Визуальное тестирование | При работе с UI |

**Важно**: каждый подключённый MCP занимает контекст. Держи активными только те, что нужны для текущей задачи.

### Claude Code + Sub-agents ([docs](https://code.claude.com/docs/ru/sub-agents))

Конфигурация sub-agents через YAML frontmatter:

```yaml
---
name: research-code
description: 'Search codebase for patterns, dependencies, and usage examples'
model: haiku
color: blue
---
Ты исследователь кода. Находи паттерны, зависимости, примеры использования.
Отвечай кратко и структурированно.
```

```yaml
---
name: implementer
description: 'Implement code changes following project conventions'
model: opus
color: green
---
Ты разработчик. Реализуй изменения строго по плану.
Следуй конвенциям из CLAUDE.md. Всегда запускай тесты после изменений.
```

```yaml
---
name: reviewer
description: 'Review code for bugs, security issues, and best practices'
model: sonnet
color: red
---
Ты code reviewer. Ищи баги, уязвимости, нарушения best practices.
Формат: CRITICAL / WARNING / SUGGESTION.
```

**Параллельный запуск**: явно указывай "запусти эти задачи параллельно" -- по умолчанию агент работает последовательно.

### Claude Code + Skills ([docs](https://code.claude.com/docs/ru/skills) | [gend.co guide](https://www.gend.co/blog/claude-skills-claude-md-guide))

Skills -- это "ленивые инструкции", загружаемые по требованию:

```markdown
# Skill: prisma-migration
Создание и применение Prisma-миграций:
1. Сначала обнови schema.prisma
2. pnpm prisma migrate dev --name описание
3. Проверь сгенерированный SQL
4. Если нужна data migration -- создай отдельный скрипт
5. Протестируй rollback: pnpm prisma migrate reset
```

Skills полезны для задач, которые нужны не каждую сессию, но требуют специфичных инструкций.

### Claude Code + Git hooks

Используй hooks для автоматизации:
- **Pre-commit hook**: автоформатирование + lint
- **Post-save hook**: прогон typecheck после каждого сохранения
- **Pre-push hook**: полный тестовый прогон

Hooks не занимают контекст агента -- они работают детерминистически.

## 7. Anti-patterns -- что НЕ надо делать

### 7.1 "Сделай всё сам"

**Проблема**: "Вот ТЗ на 3 страницы, реализуй полностью."

**Почему плохо**: агент деградирует на длинных сессиях. Первые файлы будут хорошие, последние -- мусор. Без промежуточных коммитов откатиться невозможно.

**Решение**: декомпозиция + коммит после каждого шага.

### 7.2 Отсутствие коммитов

**Проблема**: работаешь час без единого коммита.

**Почему плохо**: агент "хорошо начинает, плохо заканчивает". Если что-то пошло не так на 40-й минуте -- потеряешь хорошую работу первых 30 минут.

**Решение**: `git commit` после каждого атомарного изменения. Squash потом.

### 7.3 Никогда не чистить контекст

**Проблема**: одна сессия на весь день работы.

**Почему плохо**: контекст деградирует. Агент начинает путать файлы, забывать инструкции, повторять ошибки.

**Решение**: `/clear` после каждой завершённой подзадачи. Дешевле заново прочитать файлы, чем работать с мутным контекстом.

### 7.4 Держать все MCP-серверы включёнными

**Проблема**: 10 MCP-серверов подключены постоянно.

**Почему плохо**: описания инструментов MCP съедают до 50% контекста ещё до начала работы.

**Решение**: включай только нужные для текущей задачи. Jira нужен раз в неделю? Не держи его активным.

### 7.5 Раздутый CLAUDE.md

**Проблема**: CLAUDE.md на 500 строк с инструкциями на все случаи жизни.

**Почему плохо**: Claude начинает игнорировать инструкции. Качество следования падает линейно с количеством.

**Решение**: максимум 150-200 инструкций в корневом файле. Остальное -- в agent_docs/ и skills.

### 7.6 Copy-paste кода в промпт

**Проблема**: копируешь 200 строк кода в сообщение вместо ссылки на файл.

**Почему плохо**: тратишь контекст на то, что агент может прочитать сам. Плюс код в промпте может отличаться от актуальной версии.

**Решение**: "Прочитай файл src/services/order.ts, сфокусируйся на методе createOrder (строки 40-80)."

### 7.7 Игнорирование Plan Mode

**Проблема**: сразу просишь писать код без планирования.

**Почему плохо**: агент делает assumptions, которые не совпадают с твоими ожиданиями. Переделывать дороже, чем планировать.

**Решение**: для любой задачи сложнее однострочного фикса -- сначала план, потом реализация.

### 7.8 "Агент как Google"

**Проблема**: спрашиваешь "как работает React useEffect?" вместо того чтобы дать конкретную задачу.

**Почему плохо**: тратишь контекст на общие знания. Агент оптимизирован для работы с кодом, а не для обучения.

**Решение**: для обучения используй отдельную сессию/чат. Рабочая сессия -- для рабочих задач.

### 7.9 Не проверять output

**Проблема**: слепо принимаешь всё, что генерирует агент.

**Почему плохо**: LLM галлюцинируют. Особенно при деградации контекста. Несуществующие API, неправильные типы, subtle logic bugs.

**Решение**: review каждого изменения через `git diff`. Запускай тесты. Проверяй критические пути вручную.

### 7.10 Антропоморфизм в промптах

**Проблема**: "Ты -- гениальный мировой эксперт-архитектор с 20-летним опытом..."

**Почему плохо**: раздувает промпт бессмысленным fluff. Модель лучше реагирует на конкретные инструкции, чем на лесть.

**Решение**: вместо описания persona -- описывай task class и конкретные требования к output.
