# Wiki Manifest -- конституция проекта LLM Wiki

> Этот документ определяет все правила работы с wiki. Любой агент (Claude Code или человек), вносящий изменения, **обязан** следовать этим правилам.

---

## 1. Назначение wiki

База знаний по использованию LLM-агентов в разработке и проектной деятельности. Для себя и команды. Язык: русский с английскими терминами.

**Wiki -- это НЕ:**
- Блог или журнал (без дат, хронологии, "сегодня я попробовал...")
- Документация одного инструмента (это обзорная wiki)
- Дамп ссылок без описания

---

## 2. Структура файлов

```
LLM/
├── README.md              ← главная страница (MOC)
├── WIKI-MANIFEST.md          ← этот файл (конституция)
├── CLAUDE.md                 ← инструкции для агента
├── Инструменты/              ← то, что используем в работе
│   ├── Claude Code.md
│   ├── ...
├── Изучить/                  ← интересное, но ещё не освоенное
│   ├── Superpowers.md
│   ├── ...
└── Практики/                 ← гайды, воркфлоу, сравнения
    ├── Prompt Engineering для разработки.md
    ├── ...
```

### Папки и их смысл

| Папка | Что в ней | Когда перемещать |
|---|---|---|
| `Инструменты/` | Инструменты, которые реально используются в работе | Когда инструмент опробован и принят в стек |
| `Изучить/` | Интересные инструменты, которые стоит попробовать | При первом добавлении. Переносить в `Инструменты/` после освоения |
| `Практики/` | Гайды, воркфлоу, сравнения, принципы работы | Когда контент описывает подход/процесс, а не конкретный инструмент |

### Правило перемещения между папками

Когда инструмент из `Изучить/` становится рабочим:
1. Перемести файл в `Инструменты/`
2. Убери пометку `Статус: **Изучить**` из шапки
3. Обнови таблицу на `README.md` -- перенеси строку из секции "Изучить" в "Инструменты"
4. Wiki-ссылки `[[имя файла]]` обновлять НЕ нужно -- Obsidian резолвит по имени

## 3. Формат страницы

Каждая страница инструмента/практики **обязана** содержать:

### Шапка (обязательно)
```markdown
# Название

> Одно предложение: что это делает. ([Источник](url) | [Источник2](url2))

Назад: [[README]] | Папка: `Имя-папки/`
```

- Blockquote с кратким описанием + inline-ссылки на первоисточники
- Навигация "Назад" + указание папки
- Для файлов из `Изучить/` добавлять: `| Статус: **Изучить**`

### Тело (по ситуации)
- `## Что это` -- 1-3 предложения
- `## Ключевые возможности` или `## Возможности` -- bullet list
- `## Установка` -- команды для быстрого старта
- `## Использование` -- примеры
- `## Практические советы` -- tips из реального опыта
- Другие разделы по необходимости

### Подвал (обязательно)
```markdown
Связанные: [[Страница1]], [[Страница2]]
```

## 4. Ссылки -- правила

### Inline-ссылки на источники (КРИТИЧНО)

Каждый факт, инструкция или рекомендация должна иметь кликабельную ссылку на источник **рядом с текстом**, а не в конце файла.

**Правильно:**
```markdown
## Sub-agents ([docs](https://code.claude.com/docs/ru/sub-agents) | [Habr](https://habr.com/ru/articles/946748/))

Специализированные AI-помощники...
```

**Неправильно:**
```markdown
## Sub-agents

Специализированные AI-помощники...

---
## Источники
- https://code.claude.com/docs/ru/sub-agents
```

### Где ставить ссылки
- В заголовке раздела `## Тема ([source](url))` -- если весь раздел основан на этом источнике
- В blockquote шапки -- если это главный источник для всей страницы
- Inline в тексте `описание ([source](url))` -- для конкретных фактов

### Wiki-ссылки
- Используй `[[Имя файла]]` для связей между страницами
- Для ссылки на конкретный раздел: `[[Имя файла#Заголовок раздела]]`
- Obsidian резолвит по имени файла -- путь указывать не нужно

### Внешние ссылки
- Формат: `[Название](url)` -- всегда с человеко-читаемым текстом
- Никогда не голые URL в тексте (кроме code blocks для установки)
- Для GitHub: `[GitHub](https://github.com/org/repo)`
- Для документации: `[docs](url)`
- Для статей: `[Habr](url)`, `[DEV.to](url)` и т.д.

## 5. Главная страница (README.md)

MOC (Map of Content) -- единственная точка входа. Содержит:

1. **Таблицы** по каждой папке с колонками: Название (wiki-link), Описание, Источник (внешняя ссылка)
2. **Стек по задачам** -- комбинации инструментов для типичных сценариев
3. **Quick reference** -- ключевые команды и выбор модели

### Правила обновления главной
- При добавлении нового файла -- добавь строку в соответствующую таблицу
- При удалении файла -- убери строку
- При перемещении между папками -- перенеси строку между таблицами
- Секция "Стек по задачам" обновляется только при изменении рабочего стека

## 6. Операции

### Добавление нового инструмента
1. Создай файл в `Изучить/` (или `Инструменты/` если уже используется)
2. Заполни по формату из секции 3
3. Обязательно добавь inline-ссылку на первоисточник в шапке
4. Добавь `Связанные:` в подвале
5. Добавь строку в таблицу на `README.md`

### Добавление новой практики/гайда
1. Создай файл в `Практики/`
2. Каждый раздел с фактами -- inline-ссылка на источник
3. Добавь строку в таблицу "Практики" на `README.md`

### Обновление существующей страницы
1. Обновляй контент, сохраняя формат шапки и подвала
2. Если добавляешь информацию из нового источника -- добавь inline-ссылку
3. Если информация устарела -- обнови и пометь `*Обновлено: месяц год*` в конце файла
4. НЕ удаляй ссылки на источники при редактировании

### Удаление страницы
1. Удали файл
2. Убери строку из таблицы на `README.md`
3. Проверь `Связанные:` на других страницах -- убери мёртвые ссылки
4. Если удаляешь инструмент из стека -- обнови секцию "Стек по задачам"

### Актуализация цен и версий
1. Файл `Сравнение LLM моделей и цены` -- обновляй при значительных изменениях
2. Всегда обновляй дату в конце файла: `*Обновлено: месяц год*`
3. Inline-ссылки на pricing pages позволяют быстро проверить актуальность

## 7. Стиль текста

- **Язык**: русский, английские термины сохранять как есть (sub-agents, skills, hooks, workflow)
- **Тон**: прагматичный, без воды. "Делай X потому что Y", а не "Было бы неплохо рассмотреть возможность..."
- **Длина**: страница инструмента -- 50-150 строк. Страница практики -- до 400 строк
- **Без эмодзи**: если пользователь не просил
- **Без дат в тексте**: кроме `*Обновлено:*` пометок и данных о ценах
- **Таблицы** предпочитать спискам, где это улучшает читаемость
- **Code blocks** для команд, конфигов, примеров промптов

---

## 8. Чего НЕ делать

- Не создавать файлы вне трёх папок (Инструменты, Изучить, Практики)
- Не менять формат шапки/подвала
- Не удалять inline-ссылки на источники
- Не добавлять секции "Источники" или "Ссылки" в конец файла (всё inline)
- Не создавать вложенные подпапки внутри трёх основных папок
- Не дублировать информацию между страницами -- используй `[[wiki-ссылки]]`
- Не класть голые URL без `[описания](url)`
